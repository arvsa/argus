# Device-to-Node Assignment Bridge

## Context

Now that the frontend-v2 plan has fully shipped (hierarchy shape admin,
Node tree, live status, zones, user admin), the user tried to actually use
it end-to-end and hit a real gap: **there is no way to assign a monitored
device (IP address) to a place in the Node hierarchy, anywhere in the UI
or API.**

An Explore pass confirmed two separate problems, not one:

1. **`hierarchy.yaml` (defines the rank *shape*, e.g. Region→Site→Rack)
   has no working path into the container.** No example file ships in the
   repo, and neither `compose.yml` nor `backend/Dockerfile` mount or copy
   one in — `backend/app/seed_hierarchy.py`'s `run()` silently no-ops when
   the file is missing (`seed_hierarchy.py:74-76`). The frontend's
   `NodeTypesPage` (`frontend/src/pages/hierarchy/NodeTypes.tsx`) *does*
   let an operator build the shape by hand, one level at a time — that
   part works, just isn't documented as the actual way in today.

2. **Device↔Node assignment doesn't exist as a feature at all.** There is
   no `Device` model in the backend (the old fixed
   `Campus→Building→Room→Device` chain was fully dropped — migration
   `7488e4fab5f1`). The *only* mechanism today is hand-editing
   `pingsvc/targets.txt` with lines like `10.0.0.1,<uuid1>;<uuid2>;...`
   (see `pingsvc/cmd/pingsvc/main.go`'s `parseTargetLine`, confirmed at
   `main.go:621-646`) and restarting pingsvc — and the frontend never
   displays a Node's raw UUID anywhere, so there's no way to even get the
   ID you'd need to write into that file. `Node.path_ids` (already
   denormalized, root-first ancestor array) is exactly what the chain
   needs, just nothing computes or exposes it in this format.

The user chose to build the real bridge rather than just document the
workaround: a persisted `Device` model (address + optional Node
assignment), a CRUD API for it, an export endpoint that emits
pingsvc's exact target-file format from the DB, and a frontend UI to
manage a node's assigned devices from the Hierarchy page. `hierarchy.yaml`
also gets wired up properly as part of this pass, since it's the other
half of "how do I actually set this up."

**Naming decision** (confirmed with the user): there's already a
`/devices` page (`frontend/src/pages/Devices.tsx`) showing live ping
*status* from Redis — a different concept from this new persisted
asset+assignment record. Rather than rename/merge that existing page,
the new "assigned devices" UI lives contextually inside the Hierarchy
page's Node detail panel (where a node's breadcrumb/name already show),
not as a new standalone page. This is additive only — nothing about the
existing `/devices` ping-status page changes.

**pingsvc itself is not modified.** The bridge works entirely by making
the backend the source of truth that *exports* a file in the format
pingsvc already understands; regenerating + restarting pingsvc is a
manual/scripted step, not a live hot-reload (noted as a natural future
enhancement, out of scope here).

## Phase A — Backend: `Device` model + CRUD API

- `backend/app/models.py`: new `DeviceBase`/`DeviceCreate`/`DeviceUpdate`/
  `Device`/`DevicePublic`/`DevicesPublic`, following the exact pattern of
  `NodeType`/`Node` in the same file. Fields: `id` (UUID, server-generated),
  `addr` (str, globally unique — one row per monitored address),
  `node_id` (UUID, nullable FK to `node.id`, `ondelete="SET NULL"` — a
  deleted Node shouldn't cascade-delete device records, just orphan them;
  this differs from Node's own cascade-on-parent-delete and should be
  called out explicitly in the model's doc comment), `created_at`.
- `backend/app/crud.py`: `create_device`, matching the validation style of
  `create_node` (e.g. reject if `addr` already exists → 400, matching
  `create_user`'s duplicate-email pattern).
- New `backend/app/api/routes/devices.py`, mounted in
  `backend/app/api/main.py`'s `build_api_router` (unconditionally, not
  gated by `role` — device *assignment* is metadata relevant to both
  client and server backends, unlike the ping-pipeline routes):
  - `GET /devices/` — list, optional `?node_id=` filter (mirrors
    `nodes.py`'s `parent_id` filter style), gated by `CurrentUser`
    (read), matching `/nodes/`'s gating, since this is just hierarchy
    metadata, not a security boundary.
  - `POST /devices/`, `PATCH /devices/{id}`, `DELETE /devices/{id}` —
    **superuser-gated**, matching `node_types.py`/`nodes.py`'s write
    conventions exactly (`Depends(get_current_active_superuser)`).
  - `GET /devices/{id}` — mirror `nodes.py`'s `read_node`.
- New Alembic migration (`alembic revision --autogenerate`, from
  `backend/`) — verify the autogenerated `device` table + FK against the
  `node_type`/`node` migration's hand style
  (`9f1c2d3e4a5b_add_node_type_and_node_tables.py`) before accepting it;
  add the unique constraint on `addr` if autogenerate doesn't infer it
  from a `Field(unique=True)`.
- Tests: `backend/tests/api/routes/test_devices.py`, mirroring
  `test_nodes.py`'s structure — create/read/list/filter-by-node_id/
  update/delete, duplicate-addr rejection, superuser-gating on writes,
  and confirm deleting a `Node` sets `node_id` to `NULL` on its devices
  rather than deleting them.

## Phase B — Backend: pingsvc-format export endpoint

- `GET /devices/targets-export` in the same router (before the
  `/{id}` route, to avoid path-matching ambiguity) — **superuser-gated**
  (this reveals the full address+hierarchy map in bulk, higher
  sensitivity than a single device read). Returns `text/plain`,
  one line per device:
  - Device with `node_id` set → `{addr},{';'.join([*node.path_ids, str(node.id)])}`
    (ancestors root-first, then the assigned node itself last — matches
    the existing worked example in `development.md:91-95`,
    `127.0.0.1,acme-corp;zone-demo;rack-1`, where the last segment is the
    leaf node the device actually sits in).
  - Device with `node_id is None` → bare `{addr}` (preserves pingsvc's
    existing backward-compatible bare-address format).
  - No pagination — dump every device in one response; call this out as
    a deliberate v1 simplification if device counts ever approach
    pingsvc's target-file scale.
- Test: seed a small Node tree + a few Devices (some assigned, some not),
  assert the exact returned text body matches expectations line-for-line
  — this is the contract pingsvc's `parseTargetLine` depends on, so lock
  it precisely rather than loosely.

## Phase C — Frontend: assigned-devices UI in the Hierarchy page

- `frontend/src/api/deviceAssignments.ts` (new file — deliberately not
  reusing `api/devices.ts`, which already exports an unrelated
  `DeviceState`/`getState` for the live ping-status page): `DeviceAssignment`
  type, `getDeviceAssignments({nodeId})`, `createDeviceAssignment`,
  `updateDeviceAssignment`, `deleteDeviceAssignment`, mirroring
  `api/nodes.ts`'s shape/conventions.
- `frontend/src/pages/hierarchy/Nodes.tsx`: add an "Assigned devices"
  section to the existing detail panel (currently just
  `<NodeBreadcrumb>` + `<h2>{selected.name}</h2>`, lines 27-30) — list of
  addresses for `selected.id` with an "Add" button (`SlideOver` + form,
  address-only input) and a delete action per row (`ConfirmDialog`),
  following the exact same pattern already used for Node
  create/rename/delete in `NodeTree.tsx`. Empty state: "No devices
  assigned to this node yet."
- Tests: component test for the new section (list/add/delete), following
  the existing `NodeTree.test.tsx`/`Users.test.tsx` mocking conventions.

## Phase D — hierarchy.yaml wiring + docs

- Ship `hierarchy.yaml.example` at the repo root (mirrors the
  `.env.example` pattern) with the two worked examples already written
  in `plan/dynamic-hierarchy-multi-zone-architecture.md:97-112`.
  Instruct `cp hierarchy.yaml.example hierarchy.yaml` before first run —
  this matters because Docker auto-creates a missing bind-mount source
  as an empty *directory* (the exact bug already hit and fixed for
  `pingsvc/targets.txt` earlier this session), so the example must
  actually get copied into place, not just documented as an idea.
- Wire the bind mount into `compose.yml`'s `backend`/`prestart` services
  (`./hierarchy.yaml:/app/backend/hierarchy.yaml:ro`, optional-if-present
  — confirm Compose's behavior here matches intent, since an *always*
  mount of a required-to-exist file would break stacks that don't want
  hierarchy seeding at all; likely resolve this by keeping the mount but
  relying on the file being copied from the example, same as `.env`).
- New root-level doc, `hierarchy.md`, walking through the full loop end
  to end: (1) define the shape — either `hierarchy.yaml` + restart, or
  the frontend's `/hierarchy/types` page; (2) create Node instances via
  `/hierarchy`; (3) assign devices via the new detail-panel UI; (4)
  regenerate `pingsvc/targets.txt` via the new
  `scripts/regenerate-targets.sh` (curls `/devices/targets-export` with
  a superuser token, matching `scripts/test-auth.sh`'s `.env`-reading
  pattern) and `docker compose up pingsvc -d --force-recreate` to pick it
  up; (5) verify via the Hierarchy tree's live `NodeStatusBadge` counts.
  Replace README.md's single vague sentence about hierarchy.yaml with a
  link to this doc.
- Explicitly document as a known v1 limitation: pingsvc has no
  hot-reload, so device/hierarchy changes require this regenerate+restart
  step; a real file-watch or SIGHUP handler in pingsvc is future work.
